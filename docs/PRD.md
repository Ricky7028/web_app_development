# 專案名稱：食譜收藏夾

## 1. 專案概述
- **背景與動機**：希望能集中管理收集到的各種食譜，而且常常遇到「不知道今天要吃什麼」或「不知道怎麼運用剩下的食材」的困擾，因此需要一個系統來解決這些痛點。
- **目標用戶**：自己
- **核心價值主張**：簡單快速地收藏並搜尋食譜，不僅作為個人資料庫，更能在需要靈感時提供食譜推薦與每日驚喜，讓自炊變得有趣且方便。

## 2. 功能需求
以下為本系統的主要功能與對應的使用者故事（User Story）：

1. **新增食譜**
   - 作為自己，我希望可以輸入食譜的名稱、所需食材、步驟與照片，以便建立並完善我的食譜資料庫。
2. **搜尋食譜**
   - 作為自己，我希望可以透過輸入食譜名稱或關鍵字來搜尋，以便在想吃特定料理時能快速找到做法。
3. **根據食材推薦食譜**
   - 作為自己，我希望可以輸入目前冰箱裡有的食材，以便系統為我推薦可以使用這些食材製作的料理，避免食材浪費。
4. **個人化推薦食譜**
   - 作為自己，我希望系統能根據我收藏、搜尋或瀏覽過的紀錄，以便為我推薦我可能會有興趣嘗試的新食譜。
5. **每日食譜**
   - 作為自己，我希望每天能在系統首頁看到或收到一道「每日推薦食譜」，以便解決每天想要煮什麼的選擇障礙。

## 3. 非功能需求
- **技術限制**：使用簡單、可快速開發與部署的架構（例如：Flask + Jinja2 + SQLite），讓後續好維護。
- **效能與安全考量**：
  - 系統反應速度：搜尋或請求推薦時，期望兩秒內返回結果。
  - 基本防護：防範基本的 SQL Injection 或 XSS 攻擊，確保食譜資料不會意外毀損，且系統能穩定運作。

## 4. MVP (最小可行性產品) 範圍
如果專案時程緊迫，功能優先順序規劃如下：

- **Must Have (核心必備)**
  - 新增食譜（包含名稱、食材、步驟）
  - 搜尋食譜
  - 每日食譜（可以先使用隨機挑選的方式呈現）
- **Should Have (重要功能)**
  - 根據食材推薦食譜（可依據食材名稱進行簡單配對）
- **Nice to Have (進階加分項)**
  - 個人化推薦食譜（需要有演算法或簡單的使用者行為權重機制作為後盾）
  - 食物照片上傳與顯示
